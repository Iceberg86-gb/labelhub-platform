| Code / Contract | W/E | Introduced phase | Assumed semantic meaning |
|-----------------|-----|------------------|--------------------------|
| `submissions.status DEFAULT 'under_ai_review'` | W default | V1/M0-M1 schema | submitted answers immediately enter AI-review state |

| `SessionService.submit` | W `submitted` for session | M2-P5c | the labeling session is closed/submitted |

| `ExportIntegrationTest` fixtures | W `submitted` | M5-P3a | export tests seed rows in expected export status |
| `AiReviewIntegrationTest` fixtures | W `under_ai_review` | M3/M5 | AI-review tests seed rows in AI-review status |

## 4. Assumption Conflicts

1. **M3 AI-review assumption vs M4 reviewer queue assumption**
   - M3 path writes normal submissions as `under_ai_review`.
   - M4 default queue reads only `submitted`.
   - Result: real labeler submissions do not appear in the default reviewer queue.

2. **M3 AI-review assumption vs M5 Trusted Export assumption**
   - M3 path writes `under_ai_review`.
   - M5 export reads only `submitted`.
   - Result: real labeler submissions are omitted from Trusted Export.

3. **OpenAPI free-string submission status hides drift**
   - `Submission.status`, `OwnerSubmissionSummary.status`, and `ReviewerSubmissionSummary.status` are `string`, not an enum.
   - Generated clients cannot catch invalid or inconsistent status values.

4. **Tests encode phase-local assumptions**
   - Session tests expect `under_ai_review`.
   - Quality Ledger and Export tests seed `submitted`.
   - Integration tests pass because fixtures bypass the real submit-to-review/export path.

## 6. Ten Decision Questions

### Q1. Should normal submit write `submitted` or `under_ai_review`?

**Grep evidence**

- `SessionService.submit` line 194 currently writes `under_ai_review`.
- V1 `submissions.status` default is `under_ai_review`.
- `ReviewerQueueService` and `SubmissionMapper.selectSubmittedByTaskOrderedById` both expect `submitted`.

**Candidates**

- **A. Write `submitted` for submissions; AI review becomes a side fact.**
  - Pros: reviewer queue/export defaults become semantically natural; submit is the source-of-truth event; easy to explain.
  - Cons: changes existing behavior and V1 default; requires bug-fix exception, V9 migration/default correction, and test updates.
- **B. Keep writing `under_ai_review`; change reviewer queue/export scopes to include it.**
  - Pros: preserves current submit behavior and V1 default.
  - Cons: keeps "submitted annotations" wording misleading; export/reviewer scopes become special-case aliases; does not clarify lifecycle.
- **C. Add a third state such as `submitted_pending_ai`.**
  - Pros: can be additive on paper.
  - Cons: over-engineers without resolving existing drift; increases UI/test burden.

**Recommendation:** A. Treat submit as `submitted`; AI review/provenance is appended fact data, not the submission's primary lifecycle.

### Q2. Should `under_ai_review` remain a legal submission status?

**Grep evidence**

- Existing tests and fixtures seed `under_ai_review`.
- OpenAPI response description says submission is "queued for AI review" but status is not enum-defined.

**Candidates**

- **A. Deprecate `under_ai_review`; migrate existing rows to `submitted`.**
  - Pros: simplest lifecycle; removes ambiguous state.
  - Cons: V9 data migration touches existing dev data; needs rollback plan.
- **B. Keep `under_ai_review` as a legacy/read-compatible status but stop new writes.**
  - Pros: safer for existing rows; M6-P1 can normalize readers while V9 gradually repairs data/default.
  - Cons: two statuses may coexist temporarily.
- **C. Keep it as active status, but define it as "AI review in progress".**
  - Pros: useful if async AI review becomes real workflow state.
  - Cons: current AI review is synchronous/idempotent and does not need a durable submission status.

**Recommendation:** B for implementation safety, with a path to A once V9 migration is verified.

### Q3. What should reviewer default queue scope be?

**Grep evidence**

- Current backend and frontend default to `submitted`.
- Actual submit path produces `under_ai_review`.

**Candidates**

- **A. Default queue shows `submitted` only after submit behavior is fixed.**
  - Pros: clean semantics; small long-term surface.
  - Cons: requires Q1/Q2 fix first.
- **B. Default queue shows `submitted OR under_ai_review`.**
  - Pros: quick compatibility.
  - Cons: hides lifecycle drift and complicates count/page SQL.
- **C. Default queue removes status filter and relies only on verdict.**
  - Pros: broad reviewer visibility.
  - Cons: may show future archived/returned statuses unintentionally.

**Recommendation:** A long-term; B only as a transitional compatibility guard if V9 cannot migrate existing rows immediately.

### Q4. What is Trusted Export's submission scope?

**Grep evidence**

- `SubmissionMapper.selectSubmittedByTaskOrderedById` reads `status = 'submitted'`.
- M6-P0 export of a real labeler submission produced zero submissions because the row was `under_ai_review`.

**Candidates**

- **A. Export all final submitted answer facts, regardless of review verdict (`submitted`, approved/rejected derived by ledger).**
  - Pros: training data includes answers plus ledger/verdict context; aligns M5 artifact design.
  - Cons: needs exact status definition; if `approved/rejected` become DB statuses later, scope must be revisited.
- **B. Export only reviewer-approved submissions.**
  - Pros: "trusted" means quality-approved.
  - Cons: excludes pending data and weakens reproducibility of source-state facts; needs derived verdict query.
- **C. Export only `submitted` DB status.**
  - Pros: current SQL/simple.
  - Cons: only works if Q1 fixes submit status and no later status transitions mutate it.

**Recommendation:** A, with "trusted" coming from ledger/verdict files, not from mutating submission status.

### Q5. Can approved/rejected submissions be exported?

**Grep evidence**

- Current M4 verdict is derived from ledger, not `submissions.status`.
- `current_verdicts` is intentionally not materialized.

**Candidates**

- **A. Yes. Export includes answer facts plus ledger/verdict context for all submitted answer facts.**
  - Pros: complete training/evaluation corpus; rejected rows remain useful negative/quality examples.
  - Cons: consumers must respect verdicts.jsonl/ledger entries.
- **B. Only approved submissions.**
  - Pros: simpler downstream high-quality dataset.
  - Cons: loses audit/rejection evidence and requires filtering by derived state.
- **C. Configurable export scope.**
  - Pros: flexible.
  - Cons: extra contract/UI complexity; not needed for M6-P1 bug repair.

**Recommendation:** A for the canonical export; add filtered exports later if needed.

### Q6. Does AI review trigger a submission status transition?

**Grep evidence**

- `AiReviewService.review` appends `ai_calls`, `ai_call_in_fields`, and AI ledger entries; it does not update `submissions.status`.
- M5-P4 deliberately made AI findings ledger facts, not reviewer actions.

**Candidates**

- **A. No. AI review appends facts only.**
  - Pros: aligns idempotency and ledger semantics; avoids status churn.
  - Cons: no visible "AI in progress" status for async future.
- **B. Yes, AI review moves `submitted -> under_ai_review -> submitted/review_ready`.**
  - Pros: useful for async pipeline.
  - Cons: overfits current synchronous call and reintroduces state complexity.

**Recommendation:** A for M6-P1. If async AI review arrives, introduce a separate job/call status, not submission lifecycle mutation.

### Q7. What is the legal submission status transition table?

**Grep evidence**

- No `SubmissionStateTransitions` equivalent exists.
- Submission status is free string in OpenAPI and physical table.

**Candidates**

- **A. Minimal table: `submitted` only for immutable final answer facts; verdict is derived separately.**
  - Pros: simplest; matches append-only answer fact model.
  - Cons: no returned/resubmitted lifecycle yet.
- **B. Table: `submitted -> returned -> resubmitted` plus derived verdict.**
  - Pros: future reviewer rejection workflow.
  - Cons: beyond current product surface.
- **C. Table: `submitted -> under_ai_review -> reviewed` style.**
  - Pros: pipeline style.
  - Cons: conflicts with ledger-derived verdict design.

**Recommendation:** A for M6-P1, plus add a centralized status constants/enum boundary so future states are deliberate.

### Q8. Is `deadlineAt` required or optional?

**Grep evidence**

- OpenAPI `CreateTaskRequest.required` appears to require title/quota only; `deadlineAt` is a property but not required.
- Frontend `CreateTaskModal` has a required DatePicker rule.
- `TasksController.createTask` dereferences `request.getDeadlineAt().toLocalDateTime()` with no null guard.
- `TaskService.canPublish` requires a future deadline before publish.

**Candidates**

- **A. Required at create time. Update OpenAPI and backend validation to match frontend.**
  - Pros: publish guard simpler; UI already expects it.
  - Cons: contract change may be stricter for clients.
- **B. Optional at create time; required only before publish. Fix controller null handling.**
  - Pros: matches draft-task flexibility; publish guard already enforces before publish.
  - Cons: frontend required rule currently too strict.

**Recommendation:** B. Draft tasks should be creatable before all publish prerequisites exist.

### Q9. Can the same labeler claim the same task more than once?

**Grep evidence**

- `sessions` has unique key `(dataset_item_id, labeler_id)`, not `(task_id, labeler_id)`.
- `SessionService.claim` selects next available dataset item and creates a new session.
- M6-P0 API check created session #10 for the same labeler/task after session #9.

**Candidates**

- **A. Allow one session per dataset item per labeler.**
  - Pros: matches physical unique key and crowdsourcing tasks where one labeler labels many items.
  - Cons: UI copy should say "claim next item", not "claim task" if that is intended.
- **B. Allow one active session per task per labeler, but allow next claim after submit.**
  - Pros: prevents accidental duplicate active sessions.
  - Cons: requires query/guard; not current schema.
- **C. Allow only one session per task per labeler ever.**
  - Pros: simple quota interpretation.
  - Cons: wastes datasets with multiple items.

**Recommendation:** A, but rename/copy clarify that claim means "claim next available item".

### Q10. What regression tests are required before M6-P1 can pass?

**Grep evidence**

- Session tests currently assert submission status `under_ai_review`.
- Quality Ledger tests seed `submitted`.
- Export tests seed `submitted`.
- No integration test covers labeler submit -> default reviewer queue -> export non-zero.

**Candidates**

- **A. Minimal P0 regression set only.**
  - Tests:
    - submit returns/creates desired status,
    - submitted row appears in default reviewer queue,
    - Trusted Export includes the real submitted row.
  - Pros: tight.
  - Cons: deadline/claim bugs left outside.
- **B. Full M6-P1 regression set.**
  - Includes A plus deadline-null no-500 and duplicate-claim semantics.
  - Pros: closes all M6-P0 top findings.
  - Cons: larger phase.

**Recommendation:** B if estimated functional code remains under 500 lines; otherwise split M6-P1a status repair and M6-P1b UX/boundary repairs.

## 8. M6-P1 Implications If Recommendations Are Accepted

- Bug-fix exception required:
  - `SessionService.submit` behavior change from `under_ai_review` to `submitted`.
  - V9 migration/default correction likely needed for historical/dev rows and future direct inserts.
  - Tests expecting `under_ai_review` from submit must be updated or split into legacy-read compatibility tests.
- Additive safety layer recommended:
  - Introduce backend `SubmissionStatusCodes` constants or enum-like utility to stop string drift.
  - Add regression tests that traverse the real submit path rather than fixture-only inserts.
- OpenAPI:
  - Consider 0.10.0 adding `SubmissionStatus` enum if public clients should narrow status values.
  - If avoiding contract change in M6-P1, still document internal status semantics in decision-log.
